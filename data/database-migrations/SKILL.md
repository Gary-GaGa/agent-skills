# Database Migrations

> Every schema change is a deploy. Treat migrations with the same care you give code deploys — review, test, rollback plan.

## When to Use This Skill

- Adding, modifying, or removing columns / tables / indexes
- Deploying schema changes to production without downtime
- Choosing a migration tool
- Reviewing someone's migration for safety
- Recovering from a failed migration

---

## Core Principles

1. **Forward-only by default.** Each migration moves the schema forward. Rollbacks are separate, planned migrations.
2. **Immutable once applied.** Never edit a migration that's been run in any environment. Write a new one.
3. **One concern per migration.** Add a column = one migration. Add an index = another. Don't bundle.
4. **Every migration is reviewed.** They're code — PR, review, test in staging, then prod.
5. **Backward compatible (expand-contract).** Old app code must work with new schema during deploy.

---

## The Expand-Contract Pattern

For zero-downtime deployments:

```
Phase 1: EXPAND   — add new column/table (nullable, with defaults)
Phase 2: MIGRATE  — backfill data, deploy new app code that writes to both
Phase 3: CONTRACT — drop old column/table, remove old code
```

Each phase is a separate deploy. Never do expand + contract in one step.

### Example: Rename column `name` → `display_name`

```
Migration 1: ADD COLUMN display_name (nullable)
Deploy app: writes to BOTH name AND display_name
Migration 2: Backfill display_name FROM name WHERE display_name IS NULL
Deploy app: reads from display_name, stops writing name
Migration 3: DROP COLUMN name
```

6. **Each phase must be independently deployable and rollback-safe.**

## Safe vs Dangerous Operations

### Safe (usually no lock issues)

| Operation | Notes |
|-----------|-------|
| `ADD COLUMN` with `DEFAULT` (PG 11+) | Fast; no table rewrite |
| `ADD COLUMN` nullable without default | Instant metadata change |
| `CREATE INDEX CONCURRENTLY` | Non-blocking; takes longer |
| `CREATE TABLE` | No lock on existing tables |
| `ADD CONSTRAINT ... NOT VALID` | Doesn't scan existing rows |

### Dangerous (may lock table or cause downtime)

| Operation | Risk | Mitigation |
|-----------|------|------------|
| `ALTER COLUMN` type | Table rewrite, long lock | Add new column, migrate, drop old |
| `ADD NOT NULL` to existing column | Full table scan | Add constraint `NOT VALID`, then `VALIDATE` separately |
| `CREATE INDEX` (without CONCURRENTLY) | Blocks writes for duration | Always use `CONCURRENTLY` |
| `DROP COLUMN` | Needs `ACCESS EXCLUSIVE` lock briefly | Usually fast, but test on large tables |
| `RENAME COLUMN` | Brief lock, but breaks app code | Use expand-contract instead |
| `ALTER TABLE ... ADD FOREIGN KEY` | Locks both tables | Add `NOT VALID`, validate separately |

7. **Always use `CREATE INDEX CONCURRENTLY` on production tables.** Standard `CREATE INDEX` blocks writes.

8. **For `NOT NULL` on existing columns:**
   ```sql
   ALTER TABLE orders ADD CONSTRAINT orders_status_nn
     CHECK (status IS NOT NULL) NOT VALID;
   -- later, after backfill:
   ALTER TABLE orders VALIDATE CONSTRAINT orders_status_nn;
   ```

## Migration Tooling

| Tool | Language | Style |
|------|----------|-------|
| **golang-migrate** | Go | SQL files, up/down |
| **goose** | Go | SQL or Go functions |
| **Atlas** | Go (HCL) | Declarative (like Terraform for DB) |
| **Flyway** | Java/CLI | SQL files, versioned |
| **Alembic** | Python | Autogenerated from models |
| **Prisma Migrate** | Node | Schema-first |

### File naming

```
migrations/
├── 001_create_users.up.sql
├── 001_create_users.down.sql
├── 002_add_orders_table.up.sql
├── 002_add_orders_table.down.sql
```

9. **Sequential numbering or timestamps.** Both work; timestamps avoid merge conflicts in teams.
10. **Every `up` has a matching `down`.** Even if `down` is just a comment saying "irreversible — manual recovery needed."

---

## Rollback Strategy

11. **`down` migrations should be tested** before you need them.
12. **Some migrations are irreversible.** Dropping a column with data can't be undone. The `down` file should document this.
13. **Prefer forward-fix over rollback.** Fix the issue with a new migration rather than rolling back, when possible.
14. **Keep a runbook for each dangerous migration.** "If this fails, run X. If data is corrupted, restore from backup Y."

---

## Backfill Safely

When adding a column that needs existing rows populated:

15. **Backfill in batches, not one UPDATE.**
    ```sql
    -- BAD: locks entire table
    UPDATE orders SET display_name = name;

    -- GOOD: batched
    UPDATE orders SET display_name = name
    WHERE id BETWEEN 1 AND 10000 AND display_name IS NULL;
    -- repeat for next batch
    ```

16. **Run backfills during low-traffic periods.**
17. **Monitor lock wait times** during backfill. Kill and retry if blocking.

---

## Testing Migrations

18. **Run migrations against a copy of production data** (sanitized). Schema-only tests miss data-dependent issues.
19. **Test `up` then `down` then `up` again.** Round-trip should be clean.
20. **In CI: run all migrations from scratch** on every PR. Catches ordering and dependency issues.

## Anti-Patterns

| Anti-pattern | Fix |
|--------------|-----|
| Editing an already-applied migration | Write a new migration |
| `ALTER COLUMN` type on large table | Expand-contract pattern |
| `CREATE INDEX` without `CONCURRENTLY` in prod | Always `CONCURRENTLY` |
| Backfill in one giant `UPDATE` | Batch by primary key range |
| No `down` migration | At minimum, document "irreversible" |
| Deploying schema change + app change simultaneously | Expand-contract; decouple |
| Migration with `DROP COLUMN` before app stops using it | Remove code references first |
| No testing against real-sized data | Test with production-like volume |

---

## Checklist

Before running a migration in production:

- [ ] Migration reviewed by another engineer
- [ ] Tested in staging with production-like data volume
- [ ] Expand-contract pattern used (if column rename / type change)
- [ ] `CREATE INDEX CONCURRENTLY` for new indexes
- [ ] `NOT VALID` + separate `VALIDATE` for constraints
- [ ] Backfills are batched (not single UPDATE)
- [ ] `down` migration exists and tested (or documented as irreversible)
- [ ] Monitoring in place for lock waits and query latency
- [ ] Runbook written for failure / rollback scenario
- [ ] Old code still works with the new schema (backward compatible)
